# 架构说明

## 1. 当前系统解决什么问题

Index Journal 当前解决的是一个具体而持续的个人需求：

“每天盘后，我希望打开自己的站点，就能快速看懂美国核心指数的大致位置、阶段变化和长期表现。”

所以当前系统围绕两个替代标的构建：

- `SPY` 代表标普 500
- `QQQ` 代表纳斯达克 100

## 2. 为什么当前先用 ETF 方案

这个项目首先要保证的是：

- 数据可获取
- 维护成本可控
- 口径可持续扩展

直接处理指数本体会更快碰到授权和稳定性问题，而 ETF 替代方案在当前阶段更务实：

- Twelve Data 接入简单
- 免费额度足够
- 对个人观察用途足够接近
- 可以优先把产品形态和本地计算链路做稳

## 3. 数据链路

当前链路如下：

1. 同步脚本调用 Twelve Data `time_series`
2. 分别拉取 `SPY` 和 `QQQ`
3. 初始化时尽量补齐长期历史
4. 后续同步只刷新最近一小段时间
5. 日线以 `symbol + date` 幂等写入 `DailyPrice`
6. 服务层从数据库读取完整或指定范围的日线
7. 本地计算指标并输出给页面或 API

当前服务层负责的核心计算包括：

- 日 / 周 / 月
- `6M / 1Y / 2Y / 5Y / 10Y`
- `YTD`
- `5Y / 10Y` 年化收益
- 历史高点与回撤

## 4. 为什么只存日线，不存计算后的快照

当前选择把原始日线长期保存下来，而不是把结果固化成单独快照表。

这样做有几个直接好处：

- 计算逻辑可追溯
- 未来扩指标时不一定要改表
- 图表、统计和回撤分析都可以复用同一份基础数据

因此这个项目的数据层优先服务“长期可扩展”，而不是短期省事。

## 5. 为什么坚持本地计算指标

这个项目不依赖第三方直接返回涨跌幅，原因不是技术洁癖，而是为了保持口径稳定。

当前定义：

- 日涨跌：最新交易日 vs 前一交易日
- 周涨跌：最新交易日 vs 当前周第一个交易日
- 月涨跌：最新交易日 vs 当前月第一个交易日
- 长周期涨跌：最新交易日 vs 小于等于目标日期的最近交易日
- 年化收益：使用 CAGR
- 回撤：当前收盘价相对完整历史最高收盘价

只要这套定义掌握在本地，未来数据源切换时，页面与产品口径都不需要被迫跟着改。

## 6. 首页与日志页的角色分工

### 首页 `/`

首页是市场面板，不是后台，也不是工具箱。

它长期应聚焦于：

- 两个市场卡片
- 指标区
- 走势图
- 轻量导航

### 开发日志 `/log`

开发日志页不是补充说明页，而是承载：

- 为什么这样做
- 当前阶段的产品哲学
- AI 协作过程的沉淀
- 未来方向的边界感

这两个页面共同构成当前版本最重要的双页面结构。

## 7. 当前模块职责

### `scripts/sync-index-data.mjs`

负责：

- 请求 Twelve Data
- 回补长期历史
- 日常增量刷新
- 幂等写入 SQLite

### `lib/index-data.ts`

负责：

- 读取日线
- 统一计算首页与 API 所需指标
- 组织图表数据
- 控制 MAX 抽样等数据层细节

### `app/api/market/route.ts`

负责：

- 输出首页需要的结构化 JSON

### `app/api/market/chart/route.ts`

负责：

- 按固定 `symbol + range` 返回走势图数据
- 对外维持清晰、稳定的接口边界

### `app/page.tsx`

负责：

- 首页信息架构
- 两个市场模块的内容装配
- 空状态展示

### `app/log/page.tsx`

负责：

- 记录项目定位、设计取舍和长期方向
- 在开发态展示有限的内部状态信息

### `app/components/site-menu.tsx`

负责：

- 承载低频页面入口
- 维持轻量导航，不把首页做成复杂后台

## 8. 定时更新的边界

本项目当前只是“支持定时更新”，不是“内置调度平台”。

推荐外部调度策略：

- 北京时间每天 `06:00`
- 执行 `npm run sync:data`

这样既满足真实使用需求，也不引入过重的系统复杂度。
