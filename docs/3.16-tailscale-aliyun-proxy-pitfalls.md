# 3.16 Tailscale 与阿里云反向代理踩坑总结

## 1. 背景与目标

这次排查围绕的是一条跨三段网络的访问链路：

1. 本地 Mac 上运行 `Index Journal`，服务端口是 `3000`
2. Mac 已加入 Tailscale，Tailscale IPv4 是 `100.96.202.95`
3. 阿里云服务器已加入同一个 tailnet，Tailscale IPv4 是 `100.82.42.75`，公网 IP 是 `101.200.149.98`

目标分成两步：

1. 先让阿里云服务器能够访问本地 Mac 上的 `3000` 网站
2. 再让没有安装 Tailscale 的公司电脑，通过访问阿里云公网 `http://101.200.149.98:4000`，拿到本机 `3000` 的网页内容

最终方案是成功的，但中间踩了几层不同的问题，而且这些问题很容易互相干扰判断。

这份文档按“现象 -> 尝试 -> 为什么失败 -> 最终为什么成功”的方式记录。

---

## 2. 最终结论

最终可用链路是：

1. 本地 Mac 运行 Next 服务：`npm run dev`
2. 在 Mac 上启用 Tailscale Serve，把 tailnet 内 HTTPS 地址代理到本机 `3000`
3. 在阿里云 Nginx 中新增一个监听 `4000` 的 `server`
4. 阿里云 `4000` 再反向代理到：
   - `https://lwqdemacbook-air.tail9c9872.ts.net`

最终访问路径变成：

- 公司电脑 -> `http://101.200.149.98:4000`
- 阿里云 Nginx -> `https://lwqdemacbook-air.tail9c9872.ts.net`
- Tailscale Serve -> `http://127.0.0.1:3000`
- Next.js -> 返回 `Index Journal` 页面

---

## 3. 第一阶段问题：为什么阿里云直连不到 Mac 的 `3000`

### 3.1 初始现象

在另一台 Tailscale 服务器上：

- `ssh` 能连接到这台 Mac
- `ping 100.96.202.95` 不通
- `curl -I http://100.96.202.95:3000` 不通

这个现象一开始很容易让人产生两个误判：

1. 误以为 Tailscale 本身没通
2. 误以为本机网站可能只监听在 `127.0.0.1`

### 3.2 第一轮确认：服务有没有真正跑起来

先检查本地 Mac 的监听状态，确认：

- 最开始检查时，本地并没有网站服务在监听 `3000`
- 项目 `package.json` 中的启动脚本已经明确写了：
  - `next dev --hostname 0.0.0.0 --port 3000`
  - `next start --hostname 0.0.0.0 --port 3000`

这意味着项目配置本身不是只绑定回环地址。

把本地开发服务启动后，再检查监听状态，确认：

- `node` 正在监听 `*:3000`

这一步排除了“应用服务没起”和“只监听 `127.0.0.1`”这两种最常见问题。

### 3.3 第二轮确认：Tailscale 本身有没有通

继续检查 Tailscale 状态，确认：

- Mac 的 Tailscale 在线
- Tailnet 内其他机器在线
- `tailscale ping` 可以成功
- macOS 防火墙关闭

这一步说明：

- Tailscale 通道本身没有断
- 不是系统防火墙把所有入站都挡掉了

### 3.4 第一个误判点：`ping` 不通不代表 HTTP 一定不通

对端机器上普通 `ping 100.96.202.95` 不通。

但这件事本身不能直接推出 “网站一定访问不到”，因为：

- `ICMP ping` 是一回事
- 普通 TCP 端口连接是另一回事
- Tailscale 自己的 `TSMP ping` 又是第三回事

后来确认的实际情况是：

- `tailscale ping --tsmp 100.96.202.95` 通
- `tailscale ping --icmp 100.96.202.95` 不通

这个组合说明：

- Tailnet 路径是通的
- 但普通 `ICMP` 没有打通

这里最重要的经验是：

> `TSMP` 通只能说明 Tailscale 节点联通，不等于普通 `ICMP` 通，也不等于某个业务端口一定通。

### 3.5 第二个误判点：`ssh` 可达不等于普通 TCP 端口可达

对端机器上：

- `nc -vz 100.96.202.95 22` 成功
- `nc -vz 100.96.202.95 3000` 卡住

这一步一开始也容易误判成：

- “22 都通了，为什么 3000 不通，应该是应用的问题”

但后来判断应修正为：

- 这个 `22` 很可能走的是 Tailscale SSH 能力
- 它不能作为“普通 TCP 入站也通”的证据

经验是：

> 在启用了 Tailscale SSH 的情况下，`22` 能连上，不足以证明这台机器的任意业务端口都能被 tailnet 内其它机器直接访问。

### 3.6 第三轮确认：ACL 和本机 Shield 设置是不是问题

继续确认：

- Mac 侧 `tailscale debug prefs` 中 `ShieldsUp` 是 `false`
- Tailnet policy 是全放通：
  - `src: ["*"]`
  - `dst: ["*"]`
  - `ip: ["*"]`

这一步把两条常见怀疑排除了：

1. 不是因为这台 Mac 开了“拒绝 incoming connections”
2. 不是因为 Tailscale ACL 没放行 `3000`

### 3.7 为什么最终没有继续深挖“裸连 3000 为何不通”

到这里为止，可以明确知道：

- 本机服务正常
- Tailscale 正常
- ACL 正常
- `3000` 从远端直连仍然不通

这类问题继续深挖，通常要查：

- `pf`
- 更底层的系统网络过滤
- 某些 macOS 环境下对普通端口流量的限制

但这已经超出“尽快让服务能用”的目标。

所以这里的策略调整是：

- 不再坚持让阿里云直接访问 `100.96.202.95:3000`
- 改用更符合 Tailscale 产品能力的方式：`Tailscale Serve`

这个转向是本次排查里最重要的决策点。

## 4. 第二阶段方案：为什么改用 Tailscale Serve

### 4.1 尝试思路

在本地 Mac 上启用：

- `tailscale serve --bg 3000`

目标是让 tailnet 内其它机器通过一个 `*.ts.net` 的 HTTPS 地址访问本机服务，而不是直接打本机裸端口 `3000`。

### 4.2 第一次尝试为什么失败

第一次执行 `tailscale serve --bg 3000` 时，返回：

- `Serve is not enabled on your tailnet`

这说明问题不在命令本身，而在 tailnet 管理侧还没有启用 Serve 功能。

### 4.3 修复动作

在 Tailscale 管理台启用 `Serve` 后，再执行：

```bash
tailscale serve --bg 3000
```

随后返回：

- `https://lwqdemacbook-air.tail9c9872.ts.net/`
- 代理到 `http://127.0.0.1:3000`

### 4.4 为什么这个方案成功

这个方案成功的关键在于：

1. 它不再依赖远端机器直连本机裸露的 `3000/tcp`
2. 它把访问入口变成了 Tailscale 官方支持的 tailnet 内 HTTPS 域名
3. 它天然更适合跨设备访问和后续反向代理

随后在阿里云机器上验证：

```bash
curl -I https://lwqdemacbook-air.tail9c9872.ts.net/
```

返回 `HTTP/2 200`，证明：

- 阿里云机器已经可以通过 tailnet 域名访问本机网站

至此，第一阶段目标完成。

## 5. 第三阶段问题：为什么阿里云 `4000` 不能转发出去

在阿里云上，下一步目标是：

- 让没有 Tailscale 的公司电脑访问：
  - `http://101.200.149.98:4000`
- 由阿里云 Nginx 反向代理到：
  - `https://lwqdemacbook-air.tail9c9872.ts.net`

### 5.1 最初的 Nginx 思路本身没有错

最初写的 `server` 块逻辑是正确的：

- 监听 `4000`
- `proxy_pass` 到 `https://lwqdemacbook-air.tail9c9872.ts.net`
- 设置 `proxy_ssl_server_name on`
- 设置正确的 `Host`

也就是说：

> 问题不在反向代理逻辑本身，而在 Nginx 配置是否真的被加载。

### 5.2 第三个误判点：`nginx -t` 通过，不代表新配置已生效

一开始把新配置写到了：

- `/etc/nginx/conf.d/index-journal-4000.conf`

或者：

- `/etc/nginx/sites-available/index-journal-4000`

然后执行：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

语法检查是通过的，但：

- `curl -I http://127.0.0.1:4000` 失败
- `ss -ltnp | grep 4000` 没有任何监听

这说明：

- `4000` 根本没有被 Nginx 监听

这一步最关键的经验是：

> `nginx -t` 只能证明当前主配置语法没错，不能证明你新建的那个站点文件真的被 `include` 了。

### 5.3 第四个误判点：以为 `sites-enabled` 或 `conf.d` 一定会被加载

后来查看阿里云的主配置：

```nginx
cat /etc/nginx/nginx.conf
```

才发现这台机器的 Nginx 不是 Debian/Ubuntu 默认模板，也不是标准的模块化加载写法，而是一个手写单文件主配置。

关键事实是：

- `http { ... }` 里只有：
  - `include /etc/nginx/mime.types;`
- 没有：
  - `include /etc/nginx/conf.d/*.conf;`
- 也没有：
  - `include /etc/nginx/sites-enabled/*;`

所以前面新建的所有文件虽然存在，但从来没有被 Nginx 读进去。

这就是为什么：

- `nginx -t` 通过
- 但 `4000` 永远不监听

### 5.4 额外踩的一个复制坑：软链接命令被断行

在尝试 `sites-enabled` 路径时，还额外踩了一个 shell 复制问题：

- `ln -sf ... /etc/nginx/sites-enabled/index-journal-4000`

命令中间被断开，导致生成了一个错误软链接：

- `/etc/nginx/sites-enabled/index-`

并且 `journal-4000` 被当成一条单独命令执行。

这个坑虽然不是最终主因，但会进一步干扰判断，因为它让人误以为“链接已经建好了，只是 Nginx 没反应”。

经验是：

> 通过终端远程复制长命令时，尤其带路径和 heredoc 的命令，尽量拆成一行一条执行，避免断行导致半条命令生效。

## 6. 第四阶段修复：为什么最后直接改主配置成功

### 6.1 最终修复方案

因为阿里云的 Nginx 主配置没有 `include` 站点目录，所以最终采用最直接的方案：

- 备份 `/etc/nginx/nginx.conf`
- 把原来 `80` 端口的配置保留
- 直接在 `http { ... }` 中新增一个监听 `4000` 的 `server`

新增块的核心配置如下：

```nginx
server {
    listen 4000;
    listen [::]:4000;
    server_name _;

    location / {
        proxy_pass https://lwqdemacbook-air.tail9c9872.ts.net;
        proxy_ssl_server_name on;

        proxy_set_header Host lwqdemacbook-air.tail9c9872.ts.net;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_http_version 1.1;
        proxy_read_timeout 60s;
        proxy_connect_timeout 10s;
        proxy_send_timeout 60s;
    }
}
```

### 6.2 为什么这次成功

因为这次不再依赖：

- `conf.d`
- `sites-available`
- `sites-enabled`
- 任何主配置之外的 include 机制

配置直接写进主配置文件以后，重载 Nginx，再检查：

```bash
sudo ss -ltnp | grep 4000
curl -I http://127.0.0.1:4000
```

成功后再从外部访问：

- `http://101.200.149.98:4000`

最终页面可以正常打开，说明整条链路已经打通。

## 7. 这次问题中最重要的排查顺序

如果以后再遇到类似问题，建议按下面顺序排查。

### 7.1 先确认应用服务本身

先看服务有没有起：

```bash
lsof -nP -iTCP:3000 -sTCP:LISTEN
```

再看本机能不能访问：

```bash
curl -I http://127.0.0.1:3000
```

不要一上来就怀疑网络。

### 7.2 再区分三种“通”

要明确区分：

1. `tailscale ping --tsmp` 通
2. 普通 `ping` / `icmp` 通
3. 某个具体 `tcp` 端口通

这三件事不能混为一谈。

### 7.3 不要用 `22` 端口代替业务端口判断

如果启用了 Tailscale SSH，那么：

- `22` 可达
- 不等于 `3000` 也可达

必须直接测目标端口：

```bash
nc -vz 100.96.202.95 3000
curl -I http://100.96.202.95:3000
```

### 7.4 如果“直连端口不通”，优先考虑用产品能力绕过去

本次就是典型例子：

- 继续深挖裸连 `3000` 为何不通，会进入系统网络栈排查
- 但业务目标只是“让站点能被远端访问”

所以转而使用：

- `Tailscale Serve`

这比继续追查底层限制更符合当前目标。

### 7.5 Nginx 一定要验证“是否真的监听”

新增配置后，不能只看：

```bash
sudo nginx -t
```

还必须看：

```bash
sudo ss -ltnp | grep 4000
curl -I http://127.0.0.1:4000
```

只有监听真正出现，才说明配置被加载了。

### 7.6 遇到自定义 Nginx 时，不要假设它一定 include 目录配置

很多教程默认假设：

- `/etc/nginx/conf.d/*.conf`
- `/etc/nginx/sites-enabled/*`

都会被自动加载。

但这台阿里云不是这种标准模板，而是一个手写单文件配置。

所以排查时必须直接看：

```bash
cat /etc/nginx/nginx.conf
```

不要凭经验猜。

## 9. 这次踩坑的核心教训

### 9.1 网络联通性不能只看一个信号

这次最容易误导人的地方就是：

- `ssh` 能通
- `tsmp` 能通
- `icmp` 不通
- `3000/tcp` 不通

如果不把它们拆开看，很容易一会儿怀疑 ACL，一会儿怀疑应用，一会儿怀疑 Tailscale 整体故障。

### 9.2 先保证“能用”，再追“为什么底层不完美”

这次本机裸连 `3000` 的问题没有完全追到底层原因，但不影响当前目标达成。

对于这个项目而言，更务实的路径是：

- 先用 `Tailscale Serve` 建立稳定入口
- 再用阿里云做公网代理

这样已经满足实际使用需求。

### 9.3 自定义环境里最危险的不是报错，而是“看起来没报错”

阿里云 Nginx 这段排查里最坑的点不是语法错误，而是：

- `nginx -t` 通过
- `systemctl reload nginx` 也没报错
- 但新配置其实根本没被加载

这类“静默失败”最浪费时间。

所以以后涉及 Nginx、systemd、反向代理，一定要多加一步：

- 看监听端口是否真的出现
- 看本机回环访问是否真的有响应

## 10. 后续建议

### 10.1 如果以后长期使用，优先把阿里云代理配置模块化

当前为了快速落地，直接改了主配置文件。

长期建议是把阿里云 Nginx 改回更标准的结构：

- 在 `http { ... }` 中显式加入：
  - `include /etc/nginx/conf.d/*.conf;`
- 把 `4000` 代理配置单独拆成文件

这样后续维护更轻松，也更不容易重复踩“配置文件存在但没被加载”的坑。

### 10.2 如果以后对外长期暴露，建议加一层保护

当前的 `http://101.200.149.98:4000` 是公网入口。

如果长期使用，建议至少考虑：

1. 给阿里云代理入口加 Basic Auth
2. 或只允许公司出口 IP 访问
3. 或给公网入口加域名和 HTTPS

这样更适合长期使用。

### 10.3 如果未来还想追“为什么直连 `100.96.202.95:3000` 不通”

后续可以单独排查：

1. macOS `pf` 规则
2. 更底层的系统网络过滤
3. Tailscale 与 macOS 网络栈的具体交互

但这已经是“根因深挖”工作，不是这次“先把链路打通”的必做项。
