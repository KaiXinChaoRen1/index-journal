# AGENTS.md

本文件是 Index Journal 的 Agent 协作说明，也给人类维护者提供一份更偏“接手口径”的项目地图。

它和 `README.md` 的分工不同：

- `README.md` 解释项目是什么、为什么值得做
- `AGENTS.md` 解释接手时应该怎么理解它、改动时应遵守什么原则

## 1. 项目主名与中文表达

- 英文主名统一为 `Index Journal`
- 中文展示名可以根据页面语境使用：
  - `指数日志`
  - `市场指数监控面板`
  - `指数观察日志`

不要机械地把所有中文文案都替换成英文，但涉及项目主名时，优先保持 `Index Journal` 一致。

## 2. 这个项目到底是什么

Index Journal 不是单一工具，也不是一次性 demo。它同时承载四层意义：

1. 它是一份真实的 AI 协作开发样本  
   目标不仅是做出结果，也要保留“如何和 agent 一起把想法变成产品”的过程。

2. 它是一份新技术学习样本  
   这个项目刻意使用工作主栈之外的技术，代码本身也是学习对象。

3. 它是一份真实会被使用的个人工具  
   作者长期关注指数投资，需要一个能每天打开来看的市场面板。

4. 它是一份会长期积累的数字作品  
   未来会逐步承载产品日志、设计哲学、投资阅读和市场思考。

后续任何功能、文档或页面改动，都应该先判断它是否服务这四层意义中的至少一层。

## 3. 当前已实现范围

当前稳定范围如下：

- 数据源：Twelve Data
- 标的映射：
  - `SPY` -> S&P 500 替代追踪
  - `QQQ` -> Nasdaq 100 替代追踪
- 数据粒度：日线
- 数据存储：SQLite + Prisma
- 当前页面：
  - `/` 首页市场面板
  - `/log` 开发日志
  - `/forex` 汇率观察
  - `/btc` BTC 观察
  - `/cn-funds` 国内场内基金（本地持久化季报跟踪页）
  - `/otc-funds` 场外基金（本地持久化季报跟踪页）
- 当前接口：
  - `/api/market`
  - `/api/market/chart`
  - `/api/forex`
  - `/api/forex/chart`
  - `/api/btc`
  - `/api/btc/chart`
  - `/api/cn-funds/quarterly`
  - `/api/otc-funds/quarterly`

当前已计算指标：

- 日 / 周 / 月
- `6M / 1Y / 2Y / 3Y / 5Y / 10Y`
- `YTD`
- `5Y / 10Y` 年化收益
- 距历史高点回撤
- 历史最大回撤

当前已完成的产品取舍：

- 首页保持克制，不做工具面板式堆叠
- 图表只做趋势表达，不做复杂交易终端
- API / 环境变量状态默认不出现在首页主视图
- 低频但有价值的内容通过菜单进入

## 4. 核心开发原则

### 4.1 不为了堆功能而开发

这个项目要的是完成度，不是按钮数量。

新增功能前先问四个问题：

1. 它是否服务项目主线？
2. 它是否会破坏首页清晰度？
3. 它是否值得现在做？
4. 它是否为未来扩展打下更稳的基础？

只要这四个问题里有两个以上答案不明确，就先不要做。

### 4.2 首页必须克制

“按钮越多越像工具，而不像产品” 是这个项目的重要判断标准。

首页应长期聚焦于：

- 页面标题与说明
- 两个市场卡片
- 指标区
- 走势图
- 轻量导航

低频但有价值的内容，比如开发日志、未来的投资阅读，应该通过导航进入，而不是塞进首页主区域。

### 4.3 保留设计判断，而不是只保留代码

这个项目希望成为一个可以复盘的 AI 协作案例。

因此不要只留下“功能做完了”的结果，还要尽量留下：

- 为什么这样设计
- 为什么现在不做别的
- 这次改动服务了哪条主线

这类信息优先落在：

- `README.md`
- `AGENTS.md`
- `/log` 页面
- 必要的代码注释

### 4.4 不把个人项目做成企业级架构

当前明确不做：

- 微服务
- 前后端分离工程拆分
- MQ / Redis
- 实时 WebSocket
- 后台权限系统
- 复杂调度平台

先把“盘后更新一次、打开就能看、长期能维护”做好。

## 5. 代码与注释要求

这是学习型项目，代码不只要让 AI 写得出来，还要让人看得懂。

### 5.1 关键路径必须可读

对于以下位置，优先保证命名清晰、职责明确，并在必要时加中文注释：

- 数据同步脚本
- 指标计算函数
- 图表数据服务
- 重要页面组件
- 导航与信息架构相关组件

### 5.2 注释写“为什么”，少写废话

不要把每一行代码翻译成中文。

优先注释这些内容：

- 业务口径
- 时间与交易日边界
- 为什么使用某种实现而不是更短的写法
- 某段代码是否服务于产品约束，而不只是技术实现

### 5.3 可读性优先于炫技巧

当实现稍微复杂时，优先选择：

- 稍微啰嗦但更容易读懂的结构
- 清楚的命名
- 明确的职责边界

不要为了“写得短”牺牲可理解性。

## 6. 当前目录职责

- `app/`
  页面与 API Route。页面层尽量只负责展示，不承担复杂计算。
- `lib/`
  服务层、格式化函数、共享类型。指标计算优先集中在这里。
- `scripts/`
  数据同步脚本。负责请求数据源并写入数据库。
- `prisma/`
  数据模型定义。
- `docs/`
  面向实现和维护的补充文档。

## 7. 数据与实现约束

### 7.1 当前数据策略

- 正式统计口径只使用日线
- 指标全部在本地计算
- 数据库保存长期历史，而不是只保留最近几十天
- 日常同步优先走增量或幂等更新
- 首页指数头部采用双轨口径：
  - `昨夜收盘快照`（早晨优先）
  - `官方EOD`（中午后完成后替换）
- 页面支持手动快照刷新，但禁止自动高频轮询
- 手动快照按页面数据组做 5 分钟节流，跨用户共享同一份结果
- 手动快照与正式日线分层存储，不进入长期指标计算
- 基金季报页默认只读取本地已保存结果；只有手动输入代码或主动刷新某只基金时才会重新抓取并解析
- 时段策略：
  - BTC：允许 7x24 手动刷新
  - 指数 / 汇率：仅纽约常规交易时段允许手动刷新，周末与非交易时段直接提示不可刷新

### 7.2 不要依赖第三方直接给涨跌结果

日 / 周 / 月 / 长周期收益、年化收益、回撤都应继续从本地日线计算。

这样做的好处：

- 口径稳定
- 更容易测试
- 将来切换数据源时更容易复用页面和服务层

### 7.3 API Key 与开发态信息

- `TWELVE_DATA_API_KEY` 必须来自环境变量
- 不要硬编码到代码或文档示例之外的位置
- 开发态信息可以保留，但应隐藏在开发态或更深层页面，不要打断产品首页
- 首页和汇率页时间展示保留原始时区信息，不做强制时区换算

## 8.1 提交与回滚习惯

这是一个适合持续迭代的个人项目，也经常会由 Agent 连续修改多处文件。

因此建议遵守这个习惯：

- 只要完成了一轮相对完整、可验证的较大改动，就检查是否适合立即提交；默认应尽量在当轮结束前形成一个可回滚提交
- 提交前至少确认 `lint` 和 `build` 处于可通过状态，或明确记录当前阻塞
- 提交信息要能说明这一轮修改解决了什么，而不只是写 `update`
- 如果本轮目标是页面或交互改动，提交前应确保开发者可以直接打开网页验证，而不是还需要手动补代码、补依赖或继续收尾
- 如果本轮目标是页面或交互改动，且本地开发服务未运行，Agent 应在收尾时直接启动可访问的本地服务，并确认页面路由能打开
- 除非用户明确说“先不要提交”，否则 Agent 在完成一轮较大改动并验证通过后，应主动检查工作区并判断是否应该直接提交

这样做的目的不是追求提交数量，而是保留稳定回滚点。  
如果后续某次 AI 修改引入问题，可以更容易定位和回退到上一个可信版本。

## 9. 未来方向

这些方向是“认知上与结构上的预留”，不是现在就要全部开发：

1. 首页 / 市场面板  
   继续作为核心入口。

2. 开发日志 / 产品日志  
   继续承载设计判断、产品哲学和 AI 协作过程。

3. 投资阅读 / 市场思考  
   未来可加入短文、摘录、卡片、读书或文章后的简短评论。

4. About / 关于项目  
   可选页面，用于说明这是一个关于指数、学习和 AI 协作的个人站点。

5. 设置 / 实验区 / Debug  
   低优先级，不进入主导航核心位。

扩展时请记住：预留应该是克制的，让未来自然长出来，而不是提前摆满未完成入口。

## 10. 建议的接手顺序

如果你是新的 Agent，建议按这个顺序理解项目：

1. 先读 `README.md`
2. 再读本文件 `AGENTS.md`
3. 再看 `docs/architecture.md`
4. 然后读：
   - `app/page.tsx`
   - `app/log/page.tsx`
   - `app/components/site-menu.tsx`
   - `lib/index-data.ts`
   - `scripts/sync-index-data.mjs`
